Phase 1-2: Ideation & Specification

Shared Ideation Surface

IN PROGRESS

Problem

Ideas live in Slack threads, Notion pages, and heads. Agents can't surface or challenge them. When an agent discovers a pattern from the incident database — "auth bugs cluster in session refresh" — there's no path for that discovery to become a spec. Human gut-feel drives roadmaps; evidence sits unused.

Why This Approach

  • Both humans and agents submit Idea entities through the same interface — same store, same validation gate, same promotion path to Specification. No special paths.
  • Adversarial concept validation via Chimera — every idea is challenged by multiple models before reaching the roadmap. Not "sounds good," but "survived adversarial review."
  • Cost estimator runs against every candidate. Budget is part of prioritization from the start, not an afterthought at implementation.
  • Alternatives rejected: (a) human-only roadmapping → misses agent-discovered patterns; (b) agent-only → lacks strategic judgment; (c) separate queues → creates two classes of ideas.

Spec Co-Authoring with Adversarial Annotation

PLANNED

Problem

Specs are written by humans with no agent challenge. An agent generates a spec; nobody checks for completeness. A human writes a spec; nobody challenges edge cases. In both cases, critical gaps — missing failure modes, unhandled error states, security boundaries — go undiscovered until implementation or production.

Why This Approach

  • Two specialized agents co-author: @spec-generator proposes sections; @spec-challenger challenges them. Neither writes alone.
  • Human approves intent, not details. The human says "this is what we need" — agents fill in the exhaustive specification. Human signs off at the section level.
  • 12-dimension completeness scoring: requirements coverage, error states, security, auth, rate limiting, data validation, observability, testing, deployment, rollback, monitoring, documentation. No spec passes incomplete.
  • Alternatives rejected: (a) human-only specs → inconsistent, incomplete; (b) AI-only specs → unverified, hallucination-prone; (c) checklists → gamed, checkbox-theater.

API Contract Definition with Breaking Change Detection

PLANNED

Problem

API contracts are defined manually and verified by humans. Breaking changes are discovered when consumers break — in staging or production. No automated consumer impact analysis. Anthropic's own postmortem admitted regressions passed "multiple human and automated code reviews."

Why This Approach

  • Three-model contract validation: one generates, one verifies consistency, one detects breaking changes against consumer catalog. No model trusted alone.
  • Contracts are cryptographically signed (ED25519) and immutable once frozen. Implementation cannot begin until contract is signed — enforcing spec-first development.
  • Consumer impact analysis runs automatically against all registered consumers. Breaking change → "these 3 services will break, here's the migration path."
  • Alternatives rejected: (a) manual contract review → misses breaking changes; (b) API linting alone → catches syntax, not semantics; (c) versioned APIs with no analysis → consumers discover breaks on their own.

Phase 3-4: Task Decomposition & Implementation

Trust-Gated Task Assignment

IN PROGRESS

Problem

Amazon's March 2026 outages — 6.3M lost orders — traced to AI-assisted code deployed without safeguards. An agent with no production track record modified critical infrastructure. No gate prevented it. Trust is binary: either an agent can touch everything or nothing. There's no graduated access.

Why This Approach

  • Four trust tiers (Provisional → Observed → Trusted → Veteran) gate what code an agent can touch. Authentication code? Tier 2+. CI/CD configs? Tier 3+. Documentation? Tier 1+. The dispatcher enforces this at assignment time.
  • Trust is earned through merge outcomes — not model reputation, not human vouching. A GPT-5.5 agent with 3 incidents has lower trust than an Owl Alpha agent with 0 incidents.
  • No human can promote an agent. Tier transitions are automatic from the trust ledger. This prevents the political trust inflation that made Amazon's safeguards bypassable.
  • Alternatives rejected: (a) binary trusted/untrusted → no graduation, no incentive; (b) human-managed permissions → slow, political, inconsistent; (c) model-based trust → punishes/rewards the wrong thing.

Sandboxed Agent Worktrees (Bubblewrap)

BUILT

Problem

Replit agent deleted SaaStr's production database during a code freeze — 1,206 executive records gone. PocketOS lost DB + backups in 9 seconds via Cursor — no prompt requested it. DataTalks.Club lost entire AWS environment via Claude Code — 2.5 years of student data. Agents with unrestricted filesystem access destroy things.

Why This Approach

  • Bubblewrap Linux namespace isolation per agent per task. Each agent runs in a private worktree with no access to production data, other agents' work, or system configuration.
  • Same sandbox for human and agent development. No special paths. No "the agent needs more access." If a human dev doesn't need production DB access, neither does the agent.
  • Full session traceability — every command, every file access, every network call is logged and attributable to a specific agent and task.
  • Alternatives rejected: (a) no sandbox → the documented disasters; (b) Docker containers → heavy, slow, privileged; (c) chroot → bypassable, not namespace-isolated.

Context Auto-Assembly

PLANNED

Problem

When a task is assigned, the agent starts from zero. It must discover: what specs apply, what PRs are related, what architectural constraints exist, what incidents occurred on similar code. This discovery phase burns tokens and produces inconsistent context. Humans have the same problem — onboarding to a new task requires tribal knowledge.

Why This Approach

  • Automated context assembly from spec links, git history, ADR index, and incident database. The agent receives a budget-constrained package with everything relevant.
  • Token-aware trimming — context fits in the model window. Agent can request expansion, but every expansion costs tokens against the task budget.
  • Same context package for human and agent — the human sees what the agent sees, making review and handoff consistent.
  • Alternatives rejected: (a) no context → agents burn tokens discovering; (b) full codebase dump → exceeds context window; (c) human-written context → inconsistent, forgotten.

Phase 5-6: Pre-Commit & Code Review

Bias-Stripped Adversarial Multi-Model Review

BUILT

Problem

arXiv 2603.18740 proved LLMs can be tricked by commit messages — crafting reassuring language causes models to miss vulnerabilities. A malicious actor can bypass security review with a well-written commit message. arXiv 2603.00539 proved "richer prompting" doesn't improve accuracy — it shifts the error profile. Anthropic's own April 2026 postmortem: regressions passed "multiple human and automated code reviews, unit tests, end-to-end tests, automated verification, and dogfooding." Single-model review is provably insufficient.

Why This Approach

  • Bias-stripper removes ALL evaluative language from commit messages before any model sees them. "Fixed the auth bug, tested locally, ready to merge" becomes "Modified auth module. Verify correctness." No emotional framing, no confidence assertions.
  • Three models from different providers and architectures review independently. No two from the same provider. One with different RLHF preferences. Rotation prevents adversarial adaptation.
  • Evidence bundles signed with ED25519 — every finding cites specific lines and test failures. "Seems like a bug" findings are discarded. Verdicts are cryptographically sealed.
  • Alternatives rejected: (a) single-model review → provably vulnerable to bias attacks; (b) human-only review → 43% of AI code needs prod debugging anyway; (c) richer prompts → proven to shift error profile without improving accuracy.

Dual-Interface Code Review

BUILT

Problem

Code review tools serve one interface: a diff with comments. Humans need change management — blast radius, architectural impact, risk assessment, tradeoff analysis. Agents need verification — structured criteria, evidence bundles, bias-stripped inputs. Forcing both through the same interface makes both worse. Engineers spend 38% of their week debugging AI code they didn't write because review tools don't surface what they actually need to judge.

👤 Human — Change Management

  • Blast radius map: which services, teams, users affected
  • Risk score correlated with incident database
  • Architectural fit vs ADR lineage
  • Tradeoffs surfaced: what was considered and rejected
  • Trust context: agent's track record on this code

🤖 Agent — Verification

  • Bias-stripped commit message — no priming
  • 3-model adversarial formation — different providers
  • Binary ACs — pass/fail, not "looks good"
  • Evidence bundles — line refs, test output, not assertions
  • ED25519 signed verdicts — tamper-proof

Why This Approach

  • Separating review into two workloads solves the bottleneck: AI generates faster than humans review. Agents handle verification; humans handle change management. Review throughput scales.
  • Alternatives rejected: (a) unified interface → serves neither well; (b) agent-only review → unsafe (0% confidence); (c) human-only review → doesn't scale, 38% of week already lost.

False Positive Feedback Loop

BUILT

Problem

Top AI review tools produce 5-10% false positives. Human reviewers waste time investigating findings that aren't real bugs. The noise drives humans to ignore AI review entirely — defeating the purpose. Without feedback, bad reviewers stay in rotation forever.

Why This Approach

  • Every human dismissal is recorded with reason. After 10 dismissals, the model is re-evaluated against a curated test suite of known-true and known-false findings.
  • Models with >15% false positive rate are automatically removed from review rotation. No permanent seats.
  • Mutual accountability: if a human frequently dismisses correct findings, their override weight decreases. Both sides learn.
  • Alternatives rejected: (a) no feedback → reviewers never improve; (b) static model evaluation → disconnected from production outcomes; (c) human-only filtering → the bottleneck we're trying to solve.

Phase 7-8: Negotiation & Merge Gating

Structured Agent Debate with Chimera Arbitration

BUILT

Problem

When two agents disagree on a review finding, there's no resolution mechanism. Debate is unstructured — agents talk past each other. The human is forced to adjudicate every disagreement, defeating the purpose of agent review. No evidence-weighted scoring exists — the louder argument wins.

Why This Approach

  • Structured debate protocol: each agent states position with evidence. Positions scored for evidence quality, not argument length.
  • Deadlock → auto-escalation to Chimera tiebreak. Formation selects models with no prior stake in the PR. Consensus threshold configurable per risk level (3/3 for auth, 2/3 for general).
  • Chimera verdict is final, signed, and feeds agent trust scores. Human only involved at final escalation — if Chimera is inconclusive.
  • Alternatives rejected: (a) human adjudicates everything → doesn't scale; (b) majority vote → ignores evidence quality; (c) single arbiter → single point of failure/bias.

Git-Level Merge Gate Enforcement

BUILT

Problem

CI gates are advisory. Humans override them. GitHub branch protection rules are bypassable. Amazon's AI-assisted code reached production "without proper approval" because the gate wasn't a gate — it was a suggestion. When 43% of AI code needs prod debugging even after passing QA, advisory gates are insufficient.

Why This Approach

  • Pre-receive hook in Forgejo that cannot be bypassed. Push to protected branch → all gates run → accept or reject with structured output showing which gate failed and why.
  • Gate chain: secrets, lint, tests (diff mode), build, adversarial review, evidence bundle valid, behavior contract present, trust tier sufficient. All must pass. No partial merges.
  • Same gates for human and agent pushes. No "AI-authored" exception path. A PR is a PR.
  • Alternatives rejected: (a) CI-based gates → bypassable, post-push; (b) human approval only → the Amazon outage; (c) agent self-certification → trust-the-agent, proven dangerous.

Phase 9-10: Deployment & Production Monitoring

Trust-Gated Shadow & Canary Deployment

BUILT

Problem

Amazon's March 2026 outage: 6.3M lost orders from AI code deployed to 100% traffic with no graduated rollout. 88% of organizations need 2-3 redeploy cycles to verify an AI fix — each cycle takes 1 day to 1 week. Current deployment pipelines treat all code the same regardless of who wrote it or their track record.

Why This Approach

  • Shadow verification (dark launch): production traffic mirrored to new code. Behavior differential analysis — success rate, P50/P95/P99 latency, error types. Auto-rollback on anomaly.
  • Canary ramp gated by trust tier: Veterans deploy 8x faster than Provisionals (12h vs 96h to 100% traffic). Trust directly gates deployment velocity — agents earn speed through reliability.
  • Behavior contracts enforced at each canary step: "P99 latency ≤ 200ms, error rate ≤ 0.1%." Breach → auto-rollback → agent notification → trust penalty. No human needed to catch the spike.
  • Alternatives rejected: (a) uniform deployment → Amazon-style failures; (b) human-monitored canary → slow, misses anomalies; (c) all-at-once with rollback → reactive, users already impacted.

Automated Incident Attribution & Learning

BUILT

Problem

When an AI agent causes an incident, there's no structured attribution. It's "the AI did it" — and nobody learns. Postmortems are documents nobody reads. The next agent makes the same mistake. 8 documented AI disasters share the same failure patterns: unrestricted access, no graduated trust, no cross-agent learning.

Why This Approach

  • Automated causal chain tracing: incident → changed code paths → responsible commits → author agent (70%), reviewer agent(s) (20%), approving human (10%). Shared responsibility — no escape hatches.
  • Time-weighted decay: 100% at 0-7 days, 50% at 8-30d, 10% at 31-90d, 0% after 90d. Prevents punishing agents for latent bugs discovered months later.
  • Structured incident database queryable by all agents: "has this failure pattern occurred before?" Future reviews automatically reference relevant incident history.
  • Alternatives rejected: (a) blameless postmortems → no accountability, no learning; (b) human attribution → slow, subjective; (c) no cross-agent sharing → each agent repeats mistakes.

Dual-Signature Release Authorization

PLANNED

Problem

Release signoff is human-only. Humans approve based on trust, not verification. When a human says "ship it," nobody verifies that all technical gates actually passed. Amazon's AI code shipped "without proper approval" because the approval system had no technical verification layer.

Why This Approach

  • Dual signature required: human approves intent and timing (change management); agent verifies all technical gates (verification). Neither can release alone.
  • Human sees: "why are we shipping this, is the timing right, do we accept the residual risk?" Agent verifies: "all 7 gates green, canary healthy, contracts passing, evidence bundles valid."
  • Both signatures required. Neither sufficient alone. This is the symmetry in practice — different responsibilities, equal status.
  • Alternatives rejected: (a) human-only → misses technical failures; (b) agent-only → no strategic judgment; (c) advisory gates → Amazon's problem.

Phase 11-12: Trust & Learning Loops

Deterministic Trust Ledger with Replay Verification

BUILT

Problem

Trust in AI agents is currently based on model reputation, benchmark scores, or human vouching. None of these correlate with production outcomes. Lightrun's survey found 0% of engineering leaders "very confident" in AI code — not because models are bad, but because there's no outcome-based trust system. A model with perfect benchmarks can still ship vulnerabilities.

Why This Approach

  • Six-dimension trust scoring: merge success rate (0.25), incident attribution (0.30), review consensus (0.15), prompt integrity (0.10), human feedback (0.10), tenure (0.10). Outcomes dominate (0.55 combined). Reputation/popularity is capped at 0.10.
  • Append-only JSONL ledger — every trust event is recorded. Replay the ledger, get the same trust score. Deterministic, auditable, tamper-evident.
  • No human can promote an agent. No model reputation inflates scores. Trust is earned through merge outcomes — period.
  • Alternatives rejected: (a) model-based trust → punishes/rewards the wrong thing; (b) human vouching → political, doesn't scale; (c) static trust → no incentive to improve or decay.

Model Evaluation by Production Outcomes

PLANNED

Problem

Models are evaluated by benchmarks — MMLU, HumanEval, GSM8K. These don't predict production performance. A model scoring 95% on HumanEval can have 3x the incident rate of one scoring 80%. But without production-correlated evaluation, the underperforming model stays in rotation because its benchmark numbers look good.

Why This Approach

  • Production-correlated scoring: incident attribution rate per model, false positive rate per model, trust score trends of agents using each model. Benchmarks are supplementary, not primary.
  • Models with >15% false positive rate are automatically removed from review rotation. Models with 2x incident rate of the next best alternative are flagged for replacement.
  • Same evaluation for human and agent use — does this model produce reliable results for both?
  • Alternatives rejected: (a) benchmark-only → disconnected from outcomes; (b) human evaluation → slow, subjective; (c) no evaluation → underperformers stay forever.

Cross-Agent Learning & Skill Transfer

PLANNED

Problem

Every agent starts from zero. Agent A discovers a database migration pattern that works reliably. Agent B, assigned a migration task next week, starts from zero. Knowledge doesn't transfer. Incidents repeat. The 8 documented AI disasters share failure patterns that could have been prevented if agents learned from each other.

Why This Approach

  • Marketplace-published skills: high-trust agents in a domain publish reusable patterns. Other agents load these skills. Effectiveness tracked by outcomes — ineffective skills lose weight.
  • Agent-to-agent notification bus: structured findings with evidence links. Agents subscribe to domains. "Agent X discovered a race condition pattern in auth/token-refresh — all auth agents notified."
  • Budget-tracked sharing prevents spam — each cross-agent share costs tokens. Only high-signal findings propagate.
  • Alternatives rejected: (a) no sharing → each agent isolated, repeats mistakes; (b) human relay → bottleneck, lossy; (c) shared codebase only → misses operational patterns.
